What a VPN actually is

A VPN (Virtual Private Network) creates an encrypted tunnel between two endpoints over an untrusted network (typically the internet). The core job is three things: encapsulation (wrapping your traffic in a new packet), encryption (making the payload unreadable to outsiders), and authentication (proving both sides are who they claim to be).

The "virtual" part is key — you're building a logical private link over public infrastructure. The "private" part means confidentiality, integrity, and (usually) anonymity from the transport layer's perspective.

Core distinction VPN type = the architecture — what two things are being connected, and why.
VPN protocol = the mechanism — how the tunnel is built, secured, and maintained.

Any type can use multiple protocols. The type answers "what are we connecting?"; the protocol answers "how do we build the tunnel?"

VPN Types — the architecture

1. Remote Access VPN

The most common consumer and enterprise use case. A single user's device connects to a central VPN gateway (server). All traffic is tunneled through that gateway before reaching the internet or corporate resources. Think: employee working from home connecting to office systems, or a consumer using NordVPN/Mullvad.

Client software initiates a tunnel to a VPN concentrator. The user gets an IP address from the corporate/VPN subnet. Traffic is decrypted at the gateway and forwarded onward.

2. Site-to-Site VPN

Two entire networks are connected together — neither end user does anything. The routers/firewalls at each site handle the tunnel automatically. Used heavily by enterprises linking branch offices to HQ.

The edge devices (firewalls, routers) at each site negotiate the tunnel between themselves. Traffic between the two networks is encrypted in transit but users experience it as a flat internal network.

3. Client-to-Site

Often used interchangeably with remote access, but more precisely describes a device (not a full network) connecting to a corporate gateway. The distinction is that it's one endpoint, not two networks, and it's usually policy-driven: split tunneling, MFA, device posture checks.

4. MPLS VPN (Provider-managed)

A telecom carrier provides the private WAN — you never touch the public internet. The provider's MPLS backbone separates your traffic from other customers using VRFs (Virtual Routing and Forwarding tables). Expensive but very reliable and low-latency. Common in financial services and healthcare.

5. Cloud / Mesh / SD-WAN VPN

SD-WAN and cloud-native VPNs (AWS VPN Gateway, Azure VPN Gateway, Cloudflare WARP, Tailscale). The "VPN" here is often a mesh — each node has a tunnel to every other node, and routing is handled by software rather than dedicated hardware. Tailscale builds a WireGuard mesh between all your devices automatically.

VPN Protocols — the mechanism

IPsec (Internet Protocol Security) Recommended

The gold standard for network-layer VPNs. IPsec is a suite of sub-protocols:

  • IKE (Internet Key Exchange) — handles authentication and key negotiation. IKEv1 is old and flawed; IKEv2 is what you deploy today. Phase 1: establish a secure channel (Diffie-Hellman + certificates or PSKs). Phase 2: negotiate data channel parameters.
  • ESP (Encapsulating Security Payload) — provides encryption, integrity, and anti-replay. Uses AES-GCM in modern deployments. This is the workhorse.
  • AH (Authentication Header) — provides integrity only, no encryption. Rarely used because it can't traverse NAT cleanly.

Two modes: Transport mode (encrypts only the payload, keeps original IP header) for host-to-host; Tunnel mode (wraps the entire original packet) for gateway-to-gateway — the standard VPN scenario.

Used in: StrongSwan, Cisco ASA, Palo Alto, AWS VPN Gateway, Azure VPN.

GRE (Generic Routing Encapsulation)

GRE alone provides encapsulation with zero encryption — it wraps packets in a tunnel header to route across incompatible networks or carry multicast/broadcast traffic (which IPsec can't carry natively). Almost always paired with IPsec: GRE for routing flexibility, IPsec for encryption. Written as GRE-over-IPsec or IPsec-over-GRE (order matters for how crypto is applied).

L2TP/IPsec (Layer 2 Tunneling Protocol)

L2TP creates a Layer 2 tunnel — it encapsulates PPP frames, allowing it to carry non-IP traffic and assign IPs to remote clients like a dial-up PPP session. L2TP has zero encryption by itself, so it's always paired with IPsec. The dominant Windows/mobile VPN protocol through the 2000s–2010s. Being phased out because it double-encapsulates (L2TP inside IPsec), adding overhead with no performance advantage over IKEv2/IPsec alone.

WireGuard Modern

The newest and architecturally cleanest protocol. Operates in the Linux kernel (since 5.6). Uses only ChaCha20-Poly1305 for encryption (no algorithm negotiation — fixed cipher eliminates downgrade attacks), Curve25519 for key exchange, and BLAKE2s for hashing. The codebase is ~4,000 lines vs OpenVPN's ~100,000 — dramatically smaller attack surface.

Authentication is purely public key based — both peers pre-configure each other's public keys (no certificates, no CA). Creates a virtual network interface (e.g. wg0), routing via standard kernel routing tables. Tailscale, Mullvad, and ProtonVPN all use WireGuard under the hood.

Caveat: WireGuard doesn't hide that you're using WireGuard (recognizable UDP patterns). By design, the server logs your IP indefinitely until the session expires — Mullvad patches around this.

OpenVPN Battle-tested

A mature protocol built on OpenSSL/TLS. Runs over UDP (faster, preferred) or TCP (more reliable through restrictive firewalls). Over TCP port 443 it can disguise itself as HTTPS traffic — very useful in censored environments.

Uses a certificate authority model — server has a cert, clients have certs, both verify each other. Also supports TLS authentication (an HMAC pre-shared key on top of TLS to prevent unauthorized TLS handshakes — a DoS defense).

Slower than WireGuard due to userspace implementation, though DCO (Data Channel Offload) now moves part of it into the kernel. Extremely configurable.

SSL/TLS VPN (clientless)

Not the same as OpenVPN. A clientless SSL VPN is typically a web portal — you point your browser at https://vpn.company.com, authenticate, and get access to specific internal web apps or RDP sessions through the browser. No client software needed. Cisco AnyConnect, Pulse Secure, and Citrix operate this way.

SSTP (Secure Socket Tunneling Protocol)

Microsoft's SSL-based protocol. Tunnels PPP traffic inside HTTPS (port 443), making it very firewall-friendly. Windows-native but practically a Microsoft-only ecosystem.

PPTP (Point-to-Point Tunneling Protocol) Deprecated

Do not use MS-CHAPv2, its authentication protocol, is completely broken — offline cracking in hours with commodity hardware. NSA has been known to decrypt it passively. The only reason it still exists is legacy compatibility.

Key concepts that tie it all together

Split tunneling vs full tunnel

In full tunnel mode, all traffic from the client goes through the VPN — including YouTube, Netflix, everything. In split tunnel mode, only traffic destined for specific subnets (e.g. 10.0.0.0/8) goes through the VPN; everything else exits directly to the internet. Split tunneling saves bandwidth on the VPN gateway but means your non-corporate traffic isn't protected. An attacker on your local network can still intercept non-tunneled traffic.

Perfect Forward Secrecy (PFS)

A critical property of modern VPN protocols. Each session negotiates a fresh ephemeral Diffie-Hellman key pair. If the long-term private key is ever compromised, past sessions remain protected because those session keys were never stored. IKEv2, WireGuard, and OpenVPN all support PFS. PPTP does not.

MTU and fragmentation

Adding VPN headers reduces usable packet size. A typical Ethernet MTU is 1500 bytes; IPsec ESP adds ~50–70 bytes of overhead. If you don't configure the MSS (Maximum Segment Size) correctly on the VPN interface, you'll get silent packet drops and degraded TCP performance. WireGuard handles this better than most.

NAT traversal

IPsec was designed before NAT was pervasive. AH breaks when a NAT device changes the IP header (because AH protects it). The fix is NAT-T — wrapping IPsec ESP packets inside UDP port 4500, which NAT devices can handle. IKEv2 detects NAT and switches to UDP 4500 automatically. WireGuard is UDP-native and traverses NAT cleanly by design.

Where the industry is heading Zero Trust Network Access (ZTNA) — Traditional VPNs grant broad network access once authenticated. ZTNA (Cloudflare Access, Zscaler, Palo Alto Prisma) inverts this: access is granted per-application, per-session, based on identity, device posture, and context. The "VPN" becomes a per-request proxy rather than a persistent tunnel.

Decision framework

Scenario Recommendation
Enterprise site-to-site IPsec IKEv2 — AES-256-GCM + SHA-384
Modern remote access (endpoint VPN) WireGuard or IKEv2/IPsec
Censorship circumvention / firewall bypass OpenVPN over TCP 443 or obfuscated WireGuard
Legacy Windows-only environment SSTP or IKEv2
Personal device mesh (home lab, team) Tailscale (WireGuard underneath)
Web app access without client software SSL/TLS VPN portal
Someone recommends PPTP Hard no — broken by design
Throughput order Fastest → slowest on modern hardware at the same cipher strength:

WireGuard > IPsec (hardware offload) > OpenVPN (DCO) > OpenVPN (userspace) > L2TP/IPsec

WireGuard saturates gigabit links on a Raspberry Pi. OpenVPN struggles on the same hardware.
